The 4-Tier DICOM HierarchyA PACS never organizes files like a flat folder on a desktop. It uses a rigid, four-tier relational model. Think of it like a nesting doll system:  [ Tier 1: PATIENT ]       ───> John Doe (MRN: 999888)
       │
       └── [ Tier 2: STUDY ]    ───> Jan 2026: Left Knee Injury (Study UID: 1.2.3...)
             │
             ├── [ Tier 3: SERIES ] ───> Angle A: Axial View (Series UID: 1.2.3.99...)
             │     │
             │     ├── [ Tier 4: INSTANCE ] ───> Image Slice #1 (SOP Instance UID: 1.2.3.99.1...)
             │     └── [ Tier 4: INSTANCE ] ───> Image Slice #2 (SOP Instance UID: 1.2.3.99.2...)
             │
             └── [ Tier 3: SERIES ] ───> Angle B: Sagittal View (Series UID: 1.2.3.88...)
Tier 1: The Patient (Who?)This is the top level. It holds the demographics (Patient Name, Patient ID, Date of Birth). As you noticed, this tier stays the same across every scan the patient ever gets.  Tier 2: The Study (When and Why?)This represents a single clinical event or doctor's order. If John Doe gets an MRI of his knee in January 2026, and another MRI of his brain in May 2026, those are two separate Studies.  Every single study is assigned a globally unique Study Instance UID (0020,000D).  It also includes a unique Accession Number (0008,0050), which is the billing/order tracking number generated by the hospital's scheduling system.  This is how the PACS database separates the 3 different MRI sessions. They will have different dates, different accession numbers, and completely different Study UIDs.  Tier 3: The Series (Which Angle/Setting?)Inside a single MRI session (a Study), the scanner doesn't just take one picture. It takes multiple groups of pictures. For example, the technician might run an "Axial view" sequence, then a "Sagittal view" sequence, and then a "T2-weighted contrast" sequence.  Each of these groups is a Series.  Each group gets its own unique Series Instance UID (0020,000E).  Tier 4: The Instance (The Literal File)This is the bottom of the ladder—the individual physical .dcm file resting in the storage vault.  A single Series might contain 50 or 100 individual image slices.  Every single individual slice gets its own unique SOP Instance UID (0008,0018).  How the PACS Database Tying Works Under the HoodWhen a file enters a platform like Sectra PACS, the server's database engine splits the incoming text headers into separate tables that mimic this exact hierarchy:  The Patient Table: Logs John^Doe.  The Study Table: Links to the Patient Table, but logs the unique Study Instance UID and Study Date.  The Instance Table: Links to the Study Table, logging the individual SOP Instance UID alongside a physical file path pointer (e.g., /nas/vault/2026/05/img_99.dcm).  When a doctor opens the PACS viewing station and clicks on John Doe, the software queries the SQL database. The database doesn't just bring up a massive pool of loose images. It displays a clean tree layout:  John Doe  📅 May 20, 2026 — MRI Brain (Study UID: 1.2.840...55)  📅 January 12, 2026 — MRI Knee (Study UID: 1.2.840...11)  When the doctor clicks on the May 20, 2026 study, the workstation sends a network command (C-MOVE) matching that exact Study Instance UID. The PACS looks at its internal relational pointers, pulls only the binary image bytes welded to that specific study identity, and streams them to the screen!
